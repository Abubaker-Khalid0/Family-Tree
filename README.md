# شجرة العائلة التفاعلية

مشروع ويب بسيط (HTML / CSS / JavaScript صرف، بدون أي إطار عمل أو أدوات بناء)
يعرض شجرة عائلة تفاعلية باللغة العربية، تُبنى بالكامل من ملف بيانات واحد.

## التشغيل

لا حاجة لأي تثبيت أو خادم. افتح `index.html` مباشرة في أي متصفح حديث،
أو من داخل أي محرر (Live Server ونحوه) إن أردت إعادة تحميل تلقائية أثناء التطوير.

## بنية الملفات

```
index.html     هيكل الصفحة، الرأس، أزرار التحكم، وربط الملفات
styles.css     كل التنسيقات (تصميم أبيض/أسود، الحركة، الاستجابة)
data.js        بيانات العائلة الخام فقط — لا منطق هنا إطلاقاً
helpers.js     FamilyDB: طبقة قراءة وتحقق آمنة من البيانات (لا تلمس DOM)
tree.js        FamilyTree: محرك البناء، التوسيع/الطي، رسم خطوط الربط
app.js         نقطة الدخول: تهيئة الشجرة وربط أزرار التحكم فقط
```

كل ملف مسؤول عن شيء واحد فقط، ولا يوجد أي اسم أو علاقة عائلية "مكتوبة يدوياً"
في أي مكان خارج `data.js`.

## كيف تعمل الشجرة

- **الحالة الابتدائية:** يظهر الجد الأكبر (`rootPersonId`) فقط، في الأعلى ومتمركزاً.
- **عند الضغط على شخص لديه زوج/ة واحدة أو أكثر:** يتوسّع فرعه فيُظهر بطاقة كل
  زوج/ة (بحدود متقطعة لتمييزها عن أفراد الدم بالحدود المصمتة)، ثم أبناء تلك
  الزوجة تحديداً تحتها مباشرة — الأبناء لا يُخلطون أبداً بين الزوجات.
- **عند الضغط على أي طفل:** يتوسّع فرعه الخاص هو بنفس الطريقة، والفرع الأب
  يبقى ظاهراً، فهماً بصرياً بأن هذا الفرع الجديد امتداد له.
- **الشخص الذي لا يملك زوجة أو ملاحظات:** بطاقة ساكنة تماماً، غير قابلة للنقر.
- **الشخص الذي يملك ملاحظات فقط (بدون بيانات زوج/أبناء):** يظهر رمز معلومات
  صغير ⓘ منفصل يعرض الملاحظة في نافذة صغيرة، دون أن يُنشئ فرعاً وهمياً.
- **خطوط الربط** تُرسم عبر SVG محسوبة بواسطة JavaScript (وليس CSS ثابت)، لتبقى
  دقيقة الاصطفاف مهما اختلف عدد الأبناء أو طول الأسماء أو التمرير الأفقي.

## أزرار التحكم

- **طي الكل:** يُغلق كل الفروع المفتوحة دفعة واحدة دون تغيير موضع التمرير.
- **إعادة تعيين:** نفس الطي الكامل، مع إعادة التمرير إلى بداية الشجرة.

## كيفية إضافة أشخاص جدد

كل التعديلات تتم في `data.js` فقط — لا حاجة لمس أي ملف آخر.

### 1. إضافة طفل لشخص موجود مسبقاً

أضف كائناً جديداً في مصفوفة `people`، ثم أضف معرفه (`id`) إلى `childrenIds`
الخاصة بالزوج/ة الصحيحة:

```js
// 1) عنصر الشخص الجديد
{
    id: "p034",              // معرف فريد لم يُستخدم من قبل
    name: "اسم الشخص",        // كما يُكتب تماماً، بلا اختصار
    gender: "male",          // أو "female"
    relation: "ابن",          // أو "بنت"، أو أي وصف نصي آخر
    fatherId: "p010",        // معرف الأب (اختياري لكن يُفضّل وضعه للتوثيق)
    motherId: null,
    spouses: [],              // فارغة إن لم يكن له بعد زوج/ة أو أبناء موثّقون
    notes: ""
}
```

```js
// 2) داخل بيانات الأب/الأم، أضف المعرف الجديد إلى childrenIds الخاصة بالزوجة الصحيحة
spouses: [
    {
        id: "s008",
        type: "external",
        name: "آمنة باقراس",
        label: "الزوجة",
        childrenIds: ["p030", "p031", "p032", "p033", "p034"] // أُضيف هنا
    }
]
```

**تنبيه مهم:** مصدر الحقيقة لتحديد "أبناء مَن" هو `spouse.childrenIds`،
وليس `fatherId`/`motherId` وحدهما. تأكد دائماً من إضافة المعرف الجديد إلى
مصفوفة `childrenIds` الصحيحة، وإلا فلن يظهر الطفل تحت أي فرع.

### 2. إضافة زوج/ة جديدة لشخص موجود

أضف عنصراً جديداً داخل مصفوفة `spouses` الخاصة بذلك الشخص:

```js
spouses: [
    { id: "s099", type: "external", name: "اسم الزوجة", label: "الزوجة الثالثة", childrenIds: [] }
]
```

- استخدم تسمية `label` دقيقة (الزوجة الأولى / الثانية / الثالثة...) لأنها
  تُعرض كما هي دون أي تفسير إضافي من الكود.
- إن كانت الزوجة/الزوج شخصاً موجوداً بالفعل ضمن بيانات العائلة نفسها (وليس
  خارجياً)، استخدم `type: "internal"` مع `personId` تشير لمعرفه، بدلاً من
  `name`:

  ```js
  { id: "s100", type: "internal", personId: "p017", label: "الزوج", childrenIds: [] }
  ```

  الكود يحلّ الاسم تلقائياً من بيانات ذلك الشخص، فلا داعي لتكراره.

### 3. إضافة جيل جديد بالكامل

كرر الخطوة (1) لكل شخص في الجيل الجديد، واجعل `fatherId` (أو `motherId`)
يشير إلى الجيل الذي قبله، وأضف كل معرف إلى `childrenIds` الخاصة بالزوج/ة
المناسبة. لا حدّ لعدد الأجيال أو عدد الأبناء — الشجرة تتعامل مع أي عمق أو
اتساع تلقائياً (مع تمرير أفقي عند الحاجة).

### 4. التحقق من الأخطاء

عند فتح الصفحة، افتح "أدوات المطوّر" في المتصفح (F12) وراجع تبويب Console.
إن كان هناك معرف مكرر، أو مرجع مكسور (`fatherId`/`childrenIds` يشير إلى
معرف غير موجود)، ستظهر مجموعة تحذيرات واضحة بعنوان:

```
[شجرة العائلة] تحذيرات فحص البيانات (...)
```

هذا لا يوقف الصفحة عن العمل — الشخص أو الفرع المتأثر فقط يُتجاهل بأمان مع
تسجيل السبب، بينما تستمر بقية الشجرة بالعرض الطبيعي.

## نقاط جاهزة للتوسعة المستقبلية

الكود مقسّم بشكل يسمح بإضافة لاحقة دون إعادة هيكلة:

- **بحث بالاسم:** يمكن إضافته في `app.js` عبر المرور على كل عناصر `people`
  من `FAMILY_DATA` مباشرة، ثم استدعاء دالة "توسيع المسار" (تحتاج إضافة بسيطة
  في `tree.js` تتتبع سلسلة الآباء حتى الجذر وتفتحها تباعاً).
- **لوحة تفاصيل شخص:** يمكن بناؤها كنافذة منبثقة مشابهة لآلية `note-popover`
  الموجودة حالياً، لكن بمحتوى أغنى.
- **تصدير كصورة/PDF:** بما أن الشجرة عبارة عن DOM + SVG عاديين، يمكن لاحقاً
  استخدام مكتبة تحويل DOM إلى صورة على نفس `#tree-inner`.
- **قاعدة بيانات:** يكفي استبدال `data.js` بطلب `fetch` يُرجع نفس بنية
  `FAMILY_DATA` بالضبط؛ بقية الملفات لا تحتاج أي تعديل لأنها لا تفترض أن
  البيانات محلية.
